透明物件處理 — Dual-Render 方案

Deferred Pipeline 不支援 transparency，需要特殊處理

問題

Deferred Rendering 的限制

  • G-buffer 儲存的是 world-space data（normal, albedo, depth）
  • 透明物件的 color 需要和場景中已渲染的像素做 alpha blending
  • G-buffer 無法做 alpha blending — 它只是數據儲存，不是最終顏色
  • 透明物件的先後順序會影響結果（Order-Dependent）

解決方案

  • Dual-Render：先 deferred opaque → 再 forward transparent
  • 所有 opaque 走 deferred，所有 transparent 走 forward
  • 透明物體的渲染在 lighting pass 之後、2D overlay 之前

Transparent Forward Pass

// 透明物件渲染在半透明階段 render_path_set_target("buf", NULL, NULL, GPU_CLEAR_NONE, 0, 0.0f); // depth_test = LESS, depth_write = false // blending = ALPHA_BLEND (SRC_ALPHA, ONE_MINUS_SRC_ALPHA) for each transparent_mesh (sorted back-to-front): // 使用 lit gbuffer 做背景 // forward lighting: 只算透明物件自己的 lighting // 不能受到 CSM/Point/Spot shadow（或者只受部分影響） render_path_submit_draw("mesh_transparent")

新增 Component

// 透明物件標記 typedef struct { float alpha; // 透明度 0.0-1.0 bool transparency; // 是否開啟透明渲染 bool cast_shadows; // 透明物件是否投射陰影 } comp_transparency; Tag: CompTransparent // 有此 tag 的物件走 transparent forward pass

透明類型處理

類型特性渲染方式
Alpha Fade簡單淡入淡出Standard alpha blend, 簡單
Glass / Ice有折射 + 反射需要 env map 或 screen-space refraction, 複雜
Smoke / Particle體積透明Additive / Soft blend, 不用 depth test
Water折射 + 反射 + depth需要 reflection probe + depth-aware refraction

進階：Order-Independent Transparency (OIT)

Standard transparency 是 order-dependent（必須 back-to-front 排序）。OIT 可以任意順序渲染：

方法原理缺點
Depth Peeling多層 depth peel，每層剝離一個透明層4+ passes，昂貴
Linked Lists (A-Buffer)Fragment shader 寫 linked list，最後 resolve需要 atomic counter，複雜
Weighted Sum加權混合，不需要排序對重疊區域效果差
Weighted Blended用 depth 作為權重，近似準確最常用於半透明效果

建議

1
Phase 1: 先不做透明。Deferred pipeline 先完成 opaque rendering。
2
Phase 2: 加入 CompTransparent tag + basic alpha fade forward pass。
3
Phase 3: 加入 Weighted Blended OIT，處理玻璃等複雜透明。
